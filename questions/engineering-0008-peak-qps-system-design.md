## 问题

双十一/双十二订单查询有上千万 QPS，从系统设计和大促筹备角度怎么支撑峰值流量？

## 答案

> 🤖 AI 答案 · 模型：Claude-Opus-4.8 · 日期 2026-06-21

千万级 QPS 的订单查询，核心思路是**把流量逐层拦截、让真正打到 DB 的请求降到可控量级**，并用容量评估与预案兜底。

```
用户 → CDN/静态化 → 接入层(LB/限流) → 网关(鉴权/降级)
        → 多级缓存(本地→Redis集群) → DB(读写分离+分库分表)
                              ↘ 异步(MQ削峰) ↘ 兜底/降级
```

**1. 多级缓存（最关键）**
订单查询读多写少、热点集中。本地缓存（Caffeine）扛超热点，Redis 集群扛主体读量，命中率做到 95%+ 后 DB 压力降一两个数量级。注意缓存击穿（热 key 加互斥/逻辑过期）、穿透（空值/布隆过滤器）、雪崩（过期时间打散）。

**2. 读写分离 + 分库分表**
按 user_id 或 order_id 哈希分库分表，写主读从；查询走从库集群水平扩展。分片键要覆盖主要查询维度，避免跨片扫描；非分片维度（如卖家查订单）用异构索引/ES 兜底。

**3. 限流降级熔断**
入口按用户/接口令牌桶限流，超出快速失败而非排队拖垮系统；非核心功能（推荐、详情扩展）大促期可降级；依赖异常时熔断，返回兜底数据，保住核心链路。

**4. 异步削峰**
下单等写操作用 MQ 削峰填谷，查询侧把统计、日志、风控等旁路逻辑异步化，缩短主链路 RT。

**5. 容量评估与压测**
按历史峰值×安全系数估算 QPS，反推缓存/DB/带宽容量，提前扩容（含连接池、线程池）。全链路压测打到峰值的 1.5–2 倍，找出瓶颈与单点。

**6. 大促预案**
预热缓存与连接、限流阈值与降级开关一键可调、核心指标（RT/错误率/缓存命中/DB 负载）实时大盘 + 告警，准备回滚与扩容预案，安排值班。

一句话：**缓存挡住绝大多数读，分库分表扛住剩余写读，限流降级保核心，压测预案保平稳。**

## 延伸 / 追问

**追问：缓存和 DB 不一致怎么办？**

订单状态变更要求较强一致：采用「**更新 DB → 删除缓存**」（Cache-Aside），而非更新缓存，避免并发写覆盖；删除失败用重试 + binlog（Canal）订阅做最终一致兜底。但「更新 DB → 删缓存」并非万无一失：仍存在 race window——删缓存后、DB 提交可见前，并发读可能读到旧值并把旧值回填缓存，导致短暂脏数据。治理用**延迟双删**（更新后删一次、延迟数百毫秒再删一次，覆盖回填窗口），或读路径加**版本号/逻辑过期校验**，命中旧版本即回源。对短时不一致敏感的场景再叠较短 TTL；强一致环节（如支付状态）直接读主库，不走缓存。本质是按一致性等级分级：容忍秒级延迟的走缓存，不能容忍的走主库。
