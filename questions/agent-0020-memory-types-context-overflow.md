## 问题

Agent 的记忆类型有哪些？如果上下文超过模型最大长度，通常如何解决？

## 答案

> 🤖 AI 答案 · 模型：Claude-Opus-4.8 · 日期 2026-06-22

这题分两问：先讲**记忆有哪几类**，再讲**上下文超窗怎么救**。

### 一、Agent 的记忆类型

按「时效」分**短期 / 长期**两层；长期再按认知心理学借来的三种**内容类型**细分。

```
                ┌──────────────────────────────┐
   短期/工作记忆 │  当前任务上下文（住在 context window）  │  即时、易失
                └──────────────────────────────┘
                ┌──────────────────────────────┐
   长期记忆      │  情景 Episodic：发生过的事（历史轨迹/对话）  │
   (外部存储)    │  语义 Semantic：稳定事实/知识（偏好、文档）  │  持久、需检索
                │  程序 Procedural：怎么做（技能/规则/工具用法）│
                └──────────────────────────────┘
```

- **工作记忆 / 短期记忆（Working / Short-term）**：当前任务正在用的信息——近 N 轮对话、本轮工具结果、临时变量。载体就是**上下文窗口**，零检索、即时可见，会话结束即失效。
- **情景记忆（Episodic）**：「**发生过什么**」——历史对话、过往任务轨迹、成功/失败案例。常存为带时间戳的事件，供 few-shot 复用或复盘。
- **语义记忆（Semantic）**：「**稳定的事实与知识**」——用户偏好、领域知识、外部文档。典型落在向量库 / 知识图谱，RAG 召回的就是这一类。
- **程序记忆（Procedural）**：「**怎么做事**」——技能、规则、工具调用流程，常固化进 system prompt、工具定义或可复用的 skill/工作流。

短期管「当下」，长期（情景+语义+程序）管「沉淀」，靠**写入 / 检索 / 遗忘**联动。

### 二、上下文超过模型最大长度怎么办

核心思路：**别把所有东西都塞进窗口**——能搬到外部存储就搬，必须留在窗口的就压缩、就只放相关的。常用四招，工程上叠加使用：

| 方法 | 做法 | 代价 |
| --- | --- | --- |
| **滑动窗口** | 只保留最近 N 轮，丢最旧 | 简单，但直接丢早期信息 |
| **摘要压缩** | 把滑出的历史用 LLM 滚动压成摘要 | 保留主线，但有损 |
| **向量召回(RAG)** | 历史/知识存外部，按当前 query 检索 top-k 注入 | 容量近乎无限，但依赖检索质量 |
| **分层记忆** | 短期在窗内、长期在外存，像 OS 分页换入换出 | 最完整，工程最复杂 |

实践组合拳：**系统提示 + 任务目标钉在最前** → 近 N 轮保留**原文** → 更早的滚动**摘要** → 再早的丢进**外部存储靠检索召回**；同时给工具结果做裁剪（只留有用字段），按 token 预算从低优先级、低相关度开始裁。一句话：**短期靠压缩、长期靠检索**，让有限窗口里信噪比最高，而不是硬塞到截断。

## 延伸 / 追问

**追问：摘要压缩会丢信息，怎么降低这个风险？**

三个做法：① **分级保留**——近端留原文，只对远端历史做摘要，关键决策/约束/数字单独抽成**结构化 facts** 走精确存储，不交给有损摘要；② **滚动增量摘要**——新摘要在旧摘要基础上增量更新，而非每次从零重压，减少累积失真；③ **原文可回溯**——摘要里保留指向原始记录的引用（消息 id / 外部存储 key），需要细节时再按需召回原文。本质是「**摘要管主线、外存兜细节、结构化锁硬约束**」，而不是指望一段摘要承载所有信息。
